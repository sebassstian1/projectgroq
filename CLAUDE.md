# ProjectGroq

AI суммаризатор текста. Личный инструмент для работы со сложными 
текстами высокого уровня абстракции.

## Стек

- **Backend:** Python, Flask, flask-cors, httpx, python-dotenv
- **Frontend:** HTML + CSS + JS (один файл, без фреймворков)
- **AI:** Groq API — llama-3.1-8b-instant, llama-3.3-70b-versatile
- **Хостинг:** PythonAnywhere (sebassstian1.pythonanywhere.com)
- **Код:** GitHub (github.com/sebassstian1/projectgroq)

## Переменные окружения (.env)

- `GROQ_KEY` — API-ключ Groq
- `RECAPTCHA_SECRET` — серверный ключ reCAPTCHA v2
- `SECRET_KEY` — Flask session secret (длинная случайная строка)
- `ADMIN_USER`, `ADMIN_PASS` — учётка admin
- `FRIEND_USER`, `FRIEND_PASS` — учётка friend

## Реализовано

- Авторизация (admin, friend)
- reCAPTCHA v2 двухшаговая
- Лимит 20 запросов/день на пользователя (в памяти, обнуляется при рестарте)
- Два режима модели: Quick Scan (8B), Deep Analysis (70B)
- Четыре стиля суммаризации: brief, detailed, bullets, eli5
- Markdown форматирование результата на фронте
- Кнопка копирования
- Логирование в requests.log
- Космический фон (canvas-анимация: галактики, солнце, планеты, звёзды)

## Планы (в порядке приоритета)

1. URL-парсинг — вставил ссылку, сервер достал текст, суммаризировал
2. Перевод RU ↔ EN — отдельный эндпоинт `/translate`
3. Переезд бэкенда на Railway (автодеплой из GitHub)
4. База данных (Postgres на Railway)
5. Регистрация пользователей
6. reCAPTCHA v3 (невидимая) для публичного доступа
7. Лендинг + QR для публичного доступа

## Известные проблемы

- `daily_usage` живёт в памяти — при рестарте обнуляется. Решится с БД.
- Проверить: доходит ли `style` из фронта в промпт на бэке.  
  Если нет — это баг, кнопки стилей декоративные.
- `max_tokens: 1024` может обрезать длинные суммаризации в "Подробном" режиме.
- Пароли в `.env` хранятся в открытом виде. Для двух пользователей — ок,  
  при появлении регистрации перейти на хеши (`werkzeug.security`).

## Принципы архитектурных решений

- Держать код модульным: Groq-логика, auth, rate-limit —  
  по возможности отдельно от Flask-маршрутов.
- НЕ вводить избыточные абстракции "на будущее" (YAGNI).  
  Гибкость там, где она дёшева. Жёсткость там, где она не мешает.
- При добавлении новой фичи — подумать, не пора ли вынести что-то  
  в отдельный модуль. Но не ради этого самого.
- Изменения вносить маленькими шагами, каждый шаг — отдельный коммит.

## Направления, которые могут возникнуть

Знай об этих возможных изменениях, но НЕ закладывай их в код  
заранее. Просто имей в виду при архитектурных решениях:

- Flask → FastAPI (асинхронность, типизация, автодоки)
- PythonAnywhere → Railway
- Добавление Postgres
- Добавление других LLM-провайдеров (OpenAI, Anthropic, локальные модели)
- Новые эндпоинты (`/translate`, `/parse_url`, `/notes`)

## Правила для Claude Code

- Перед правкой кода — прочитай целевой файл полностью.
- Секреты никогда не выводить в чат. Работать только через `os.getenv()`.
- Крупные изменения — сначала план, потом правка.  
  Не переписывать куски > 30 строк без согласования.
- Любую новую фичу — в отдельный коммит с осмысленным сообщением.
- Комментарии в коде — на русском, краткие, только там где это реально помогает.
- Если замечаешь баги/запахи вне задачи — отметь в "Известные проблемы",  
  но не чини без спроса.
- При сомнении в решении — задай вопрос, не додумывай.
